# Design: Writing Quality Rules Module

## Architecture Overview
Spec artifacts generated by SpecOps lack prose quality guidance — design rationales hedge with modal verbs, architecture overviews describe features instead of framing problems, and jargon accumulates without definition. This module adds a `core/writing-quality.md` file that flows through the existing three-layer pipeline (core → generator → platforms) to embed writing rules in all 4 platform outputs. The module complements `core/simplicity.md`: simplicity governs scope (what to include), writing quality governs expression (how to say it).

## Technical Decisions

### Decision 1: Standalone Core Module vs. Template Annotations
**Options Considered:**
1. New `core/writing-quality.md` module — rules embedded in generated SKILL.md as a dedicated section, applied during Phase 2
2. Inline annotations in spec templates (feature-requirements.md, design.md, etc.) — rules scattered across each template as HTML comments

**Decision:** Option 1 — standalone core module
**Rationale:** Writing rules apply uniformly across all spec types (feature, bugfix, refactor). Scattering them across templates creates duplication and maintenance burden. A single module follows the same pattern as `core/simplicity.md` and `core/error-handling.md`.

### Decision 2: No Abstract Operations
**Decision:** The module contains zero abstract operations (READ_FILE, WRITE_FILE, etc.)
**Rationale:** Writing rules are prose guidance for the agent, not file operations. This matches `core/simplicity.md` (24 lines, zero abstract ops). The generator's `substitute_tools()` function is a no-op for this module.

### Decision 3: Rule Organization by Concern
**Decision:** Six sections organized by writing concern: Structure and Order, Precision and Testability, Clarity and Conciseness, Audience Awareness, Collaborative Voice, Self-Check
**Rationale:** Each section maps to a distinct cognitive skill the agent must apply. This prevents rules from blurring together and makes validation markers specific enough to confirm substance, not just skeleton.

## Component Design

### core/writing-quality.md (~100 lines)
**Responsibility:** Define imperative writing rules for Phase 2 spec artifact generation
**Interface:** Rendered into platform outputs via `{{ writing_quality }}` template variable
**Dependencies:** None (standalone prose module)

### Generator Pipeline Changes
**Responsibility:** Wire the new module into all 4 platform outputs
**Files to Modify:**
- `generator/generate.py` — add `"writing_quality": core["writing-quality"]` to `build_common_context()` (line 348)
- `generator/templates/claude.j2` — add `{{ writing_quality }}` after `{{ simplicity }}` (line 35)
- `generator/templates/cursor.j2` — add `{{ writing_quality }}` after `{{ simplicity }}` (line 40)
- `generator/templates/codex.j2` — add `{{ writing_quality }}` after `{{ simplicity }}` (line 33)
- `generator/templates/copilot.j2` — add `{{ writing_quality }}` after `{{ simplicity }}` (line 33)

### Validation Markers
**Responsibility:** Ensure writing quality rules survive generator pipeline
**Markers:**
```python
WRITING_QUALITY_MARKERS = [
    "## Writing Quality",
    "### Structure and Order",
    "### Precision and Testability",
    "### Clarity and Conciseness",
    "### Audience Awareness",
    "### Self-Check",
    "### Sources",
    "ANT test",
    "active voice",
]
```

Added to both `validate_platform()` AND the cross-platform consistency loop (per prior decision: Gap 31).

### README Attribution
**File:** `README.md` — new "Writing Philosophy" section between "Architecture" and "Contributing" (after line 163)
**Content:** 9 attributed leaders with one-line description of their contribution to SpecOps writing quality

## Testing Strategy
- Regenerate all outputs: `python3 generator/generate.py --all`
- Validate: `python3 generator/validate.py` (checks WRITING_QUALITY_MARKERS in all platforms)
- Run full test suite: `bash scripts/run-tests.sh`
- Verify no abstract ops in module: `grep -n 'READ_FILE\|WRITE_FILE\|EDIT_FILE' core/writing-quality.md` (should be empty)
